# Language-Neutral Schema Artifacts

The audit library publishes two language-neutral artifacts that
describe the wire shape of every audit event so non-Go consumers
(SIEM rule authors, Python/Java services, compliance teams) can
validate or parse events without re-deriving the schema from Go
source.

The artifacts are generated by [`audit-gen`](code-generation.md) from
the same taxonomy YAML the library consumes. They are versioned with
each release.

## What the artifacts are

| Artifact | Purpose |
|---|---|
| `audit-event.schema.json` | JSON Schema (Draft 2020-12) that validates a single audit event JSON document emitted by `audit.JSONFormatter`. |
| `audit-event.cef.template` | Plain-text documentation describing the CEF mapping `audit.CEFFormatter` applies. SIEM rule authors read it to align field-extraction rules with the library's CEF output. |

Both artifacts are deterministic and self-contained — no external
`$ref` lookups, no online dependencies, no per-deployment state.

## Where to find them

### Per-release framework-only artifacts

Every tagged release publishes a **framework-only** version of both
artifacts (no taxonomy events; just the always-present framework
fields and the reserved standard fields). These are intended for
consumers who do not have access to the producer's taxonomy or who
build cross-deployment tooling:

```bash
gh release download v1.0.0 \
  --pattern 'audit-event.framework.schema.json' \
  --pattern 'audit-event.framework.cef.template'
```

The framework-only artifacts are regenerated on every change to the
framework field set, the reserved standard field list, or the CEF
mapping. The `make regen-schema-artifacts-check` CI guard rejects
stale artifacts on every PR.

### Per-consumer custom artifacts

Generate a schema covering your own taxonomy's custom fields with:

```bash
audit-gen -format json-schema \
  -input my_taxonomy.yaml \
  -output my-audit-event.schema.json

audit-gen -format cef-template \
  -input my_taxonomy.yaml \
  -output my-audit-event.cef.template
```

`-package` is not required when `-format` is `json-schema` or
`cef-template` — those outputs are not Go source.

## Using the JSON Schema

### Shape

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id":     "https://github.com/axonops/audit/schema/audit-event.schema.json",
  "type":    "object",
  "required": ["timestamp", "event_type", "severity", "app_name", "host", "timezone", "pid"],
  "properties": {
    "timestamp":      { "type": "string", "format": "date-time" },
    "event_type":     { "type": "string" },
    "severity":       { "type": "integer", "minimum": 0, "maximum": 10 },
    "event_category": { "type": "string" },
    "app_name":       { "type": "string" },
    "host":           { "type": "string" },
    "timezone":       { "type": "string" },
    "pid":            { "type": "integer" },
    "duration_ms":    { "type": "integer" },

    "actor_id":       { "type": "string" },
    "source_ip":      { "type": "string" },
    "...":            "(31 reserved standard fields total)"
  },
  "oneOf": [
    {
      "properties": {
        "event_type": { "type": "string", "const": "user_create" },
        "actor_id":   { "type": "string" },
        "outcome":    { "type": "string" }
      },
      "required": ["event_type", "actor_id", "outcome"]
    }
  ],
  "unevaluatedProperties": false
}
```

The `oneOf` block has one branch per event type. Each branch:

- Pins `event_type` to the specific event name via `const`.
- Lists the event's required + optional custom fields with types
  inferred from the taxonomy YAML's `type:` annotations.
- Constrains `event_category` to the categories the event belongs
  to (when applicable).

Strict unknown-property rejection is applied at the schema **root**
via `unevaluatedProperties: false`. This catches typos and uncoded
fields without over-rejecting framework or reserved properties.

### Validation in Python

```python
import json
from jsonschema import Draft202012Validator

with open("audit-event.framework.schema.json") as f:
    schema = json.load(f)

validator = Draft202012Validator(schema)

# Validate one event
with open("event.json") as f:
    event = json.load(f)

errors = sorted(validator.iter_errors(event), key=lambda e: e.path)
if errors:
    for err in errors:
        print(f"{list(err.path)}: {err.message}")
else:
    print("event valid")
```

### Validation in Java (everit-org/json-schema)

```java
import org.everit.json.schema.Schema;
import org.everit.json.schema.loader.SchemaLoader;
import org.json.JSONObject;
import org.json.JSONTokener;

import java.io.FileInputStream;

JSONObject rawSchema = new JSONObject(
    new JSONTokener(new FileInputStream("audit-event.framework.schema.json")));
Schema schema = SchemaLoader.load(rawSchema);

JSONObject event = new JSONObject(
    new JSONTokener(new FileInputStream("event.json")));
schema.validate(event); // throws ValidationException on failure
```

### Validation in TypeScript (ajv)

```typescript
import Ajv2020 from "ajv/dist/2020";
import schema from "./audit-event.framework.schema.json";

const ajv = new Ajv2020();
const validate = ajv.compile(schema);

const ok = validate(event);
if (!ok) console.error(validate.errors);
```

### SIEM rule authoring (Splunk SPL)

The schema is a useful **field reference** when writing SPL — every
property in the schema is a top-level JSON key in the event:

```spl
index=audit sourcetype=audit_json
| spath
| where event_type="user_create" AND outcome="failure"
| stats count by actor_id, source_ip
```

For a list of every property your taxonomy emits, generate the
per-consumer schema and run:

```bash
jq -r '.properties | keys[]' my-audit-event.schema.json
```

## Using the CEF template

The CEF template is a documentation artifact — not a runtime template
engine. SIEM rule authors read it to align field-extraction rules
with the library's CEF output. CEF generation in production is still
done by `audit.CEFFormatter`.

### Shape

The template lists:

1. The CEF v0 header with placeholders for vendor, product, version,
   event_type (event class id), label, and severity.
2. Framework field → CEF key mappings (`deviceProcessName`,
   `dvchost`, `dtz`, `dvcpid`, `rt`).
3. Reserved standard field → CEF key mappings (`suser`, `suid`,
   `src`, `spt`, `duser`, ...) sourced from
   `audit.DefaultCEFFieldMapping()`.
4. Per-taxonomy custom-field allocation to the CEF custom string
   slots `cs1..cs6` (with paired `cs<N>Label`).
5. Escape rules and severity translation the library applies.

### Worked example: Splunk CEF parser config

The Splunk Stream addon's CEF parser identifies fields by their CEF
extension key. To map a custom audit field to a Splunk indexed field,
look up the slot allocation in the template:

```
# Custom fields (per-event, in cs1..cs6 allocation order):
#
#   user_create
#     marker                           → cs1 (cs1Label="marker")
#     session_count                    → cs2 (cs2Label="session_count")
#     retry_after                      → cs3 (cs3Label="retry_after")
```

In Splunk's `props.conf`, you would then write:

```ini
[audit_cef]
EXTRACT-marker = cs1=(?<marker>[^ \\|]+)
EXTRACT-session = cs2=(?<session_count>\d+)
```

## Verification

Validate a real event from any example against the framework-only
schema:

```bash
go run ./examples/01-basic | head -1 > /tmp/event.json

# With Python's jsonschema CLI (pip install jsonschema)
jsonschema -i /tmp/event.json deploy/schemas/audit-event.framework.schema.json && \
  echo "framework-only schema validates the event"
```

The library's own BDD suite includes
[`schema_artifacts.feature`](../tests/bdd/features/schema_artifacts.feature)
which round-trips the generated schema against representative event
payloads on every CI run — `make test-bdd-core` exercises it.
